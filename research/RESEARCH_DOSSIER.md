## Email Address Derivation  (`derivation`)

> **Verifier net assessment:** Largely reliable dimension with two systematic weaknesses. The RFC/provider facts (RFC 2142 role accounts, RFC 5233 subaddressing timelines, Gmail's ASCII-only registration), the Hunter confidence-score anchors (95% observed / 60% pattern-guessed, confirmed from Hunter's own help center), the nicknames dataset details, and the company-size format brackets all check out against primary or independent sources — 6 of 9 claims fully VERIFIED. The weaknesses: (1) the precise distribution percentages are vendor-marketing numbers that are internally inconsistent even within Prospeo's own pages (first.last is 35% on one page, and the '38%' figure appears to be a conflation with Hunter's accept-all-domain statistic), and the '20-30 permutations cover 95%' figure is unsourced — Prospeo actually recommends 3-6 candidates and reports 3 formats covering 89%; (2) the collision-resolution claim overstates middle-initial-first as 'almost universal' when AD practice sources show numeric suffixes are at least as common and precedence is org-specific. None of the corrections invalidate the architectural implications (size-conditioned priors, shared pattern KB, narrow verification budget, collision-risk flags, role/plus canonicalization) — those all survive verification. Recommendation for mailmetero: treat all format-share percentages as tunable priors seeded from this data rather than hard-coded truths, and generate both middle-initial and numeric-suffix collision candidates without assuming an ordering.


### 1. first.last@ is the single dominant corporate email format, but its share varies enormously by data source and company size — roughly 38% of companies overall, rising to ~48-56% at 1,000+ employee firms, and ~77% of actual addresses in BounceZero's own 3,006-address audit.
- **confidence (researcher):** medium · **verdict:** (unmatched)
- **detail:** Prospeo's 2026 format guides give a cross-company distribution: first.last ~38%, first ~17%, flast ~12%, firstlast ~10%, first_last ~6%, f.last ~5%, firstl ~4%, last-name-first variants ~3% each. Size matters more than the global average: first@ dominates companies under ~50 people (42-71%), flast peaks at 51-500 employees (~42-45%), first.last standardizes at 1,000+ (48-56%). Per-company data from RocketReach/LeadIQ shows individual companies are highly monomorphic (e.g., 76-98% on one pattern once chosen). Note the address-weighted vs company-weighted distinction: BounceZero's 77% first.last figure is address-weighted and skews toward large enterprises, which is why it exceeds the ~38% …
- **→ implication for mailmetero:** mailmetero's default (no-learned-pattern) ranking should be conditioned on company headcount, not a single global prior: for <50-person companies rank first@ near the top; for mid-size promote flast@; for enterprises lead with first.last@. Store employee-count (from enrichment or user input) as a ranking feature, and keep company-weighted and address-weighted priors as separate tunable tables.

### 2. A learned per-domain pattern beats any global prior, and the industry-standard confidence model (Hunter's) is: ~90-95% score when the exact address was observed in a public source, ~60-70% when it is only pattern-derived, with score boosted by number of independent sources and mail-server verification result.
- **confidence (researcher):** medium · **verdict:** (unmatched)
- **detail:** Hunter derives one dominant pattern per domain from its index of publicly-scraped addresses, then applies that pattern to new names at the same domain; the confidence score combines (a) strength/support of the identified domain pattern, (b) SMTP-level mail server check result, and (c) count of public sources where the address appeared. Once a domain's pattern is established, all subsequent lookups on that domain get elevated confidence. Companies almost never change their pattern after IT rolls it out, so learned patterns are durable ('you're matching a template, not solving a mystery' — non-standard patterns like underscores, middle initials, numeric suffixes are the residual ~5%).
- **→ implication for mailmetero:** This is the core reason mailmetero-as-a-service beats offline BounceZero: build a shared Neon Postgres table domain_patterns(domain, pattern, observed_count, verified_count, last_seen_at) that every tenant's confirmed/verified results feed (privacy: store patterns and counts, not tenant-attributed people). Score as a blend: pattern support at domain (dominant), verification outcome, and recency; emit calibrated bands (e.g., 95+ = observed/verified, 70-90 = strong learned pattern, 50-70 = prior-only guess) so API consumers can threshold.

### 3. Generating ~20-30 permutations covers ~95% of corporate naming conventions, but verifying only the top 3 size-appropriate patterns finds the right address about 9 times out of 10 — so candidate generation should be wide but verification narrow.
- **confidence (researcher):** medium · **verdict:** CORRECTED
- **detail:** Permutator tools (Mailmeteor, Metric Sparrow, Tomba, Overloop) converge on the same canonical set: first.last, flast, first, firstlast, first_last, f.last, first.l, firstl, last.first, lastf, last, initials (fl, f.l), plus hyphen variants. Prospeo's testing reports the top-3 patterns for the target's size bracket hit ~90% of the time, and that the long tail (first_last 6%, f.last 5%, firstl 4%, last-first ~3%) is 'where most bounced guesses come from'. Since mailmetero's verification budget is constrained (port-25 blocking, external verify-API cost per check), spraying 20 candidates at a verifier is wasteful and reputation-damaging.
- **→ implication for mailmetero:** Architect derivation as two stages: (1) cheap, exhaustive ranked candidate list (return all ~25 with scores in the API response), (2) budgeted verification that probes only the top-N (default 3) unless the caller pays for deep mode. This directly bounds per-request verification-API spend and quota accounting in the multi-tenant model.
- **⚠ VERIFIER CORRECTED:** The top-3 hit rate is confirmed verbatim: Prospeo's permutator page states 'In our testing, the top 3 patterns found the right email about 9 times out of 10,' and 'Three formats — {first}@, {first}.{last}@, and {f}{last}@ — cover 89.09% of businesses.' However, the '20-30 permutations cover ~95%' figure could not be sourced anywhere; Prospeo's guess-format guide explicitly recommends generating '3-6 candidates max,' not 20-30. Also unconfirmed: the sub-claim that the long-tail patterns are 'where most bounced guesses come from' — Prospeo attributes bounce risk to catch-all domains (27% bounce on accept-all vs 1% on valid), not to specific patterns. The wide-generation/narrow-verification architecture is still sensible, but the 95% coverage number should not be quoted as fact.

### 4. Diacritics must be handled by ASCII folding, not internationalized email: despite RFC 6530 (EAI/UTF-8 local parts), Google does not allow registering non-ASCII addresses and corporate directories overwhelmingly assign ASCII local parts — but German-style transliteration (ü→ue vs ü→u) is ambiguous, so both folded variants must be generated.
- **confidence (researcher):** high · **verdict:** (unmatched)
- **detail:** RFC 6530 permits full Unicode in local parts, but support is described as unreliable/spotty in practice; Gmail can route to internationalized addresses but won't create non-ASCII mailboxes, and guidance across sources is to 'stick to ASCII transliterations for reliability'. That leaves two competing folds for European names: simple diacritic-stripping (Müller→muller, André→andre, via Unicode NFKD + strip combining marks) versus locale transliteration (Müller→mueller, Größe→groesse, ß→ss), and both occur in the wild at German/Austrian/Nordic companies. French/Spanish/Portuguese names almost always simple-strip (José→jose, François→francois).
- **→ implication for mailmetero:** mailmetero's normalizer should NFKD-strip by default and additionally emit German-transliteration variants (ue/oe/ae/ss) as separate ranked candidates when the name or company TLD (.de/.at/.ch) suggests Germanic origin; never emit raw UTF-8 local parts as primary candidates. Record which fold won per domain in the pattern KB so the second lookup at the same German company gets it right immediately.

### 5. Apostrophes and hyphens in surnames have no universal corporate convention; the most common treatment is stripping the punctuation entirely (O'Brien→obrien, Smith-Jones→smithjones), with hyphen-preserved a real minority variant and apostrophe-preserved technically legal but rare.
- **confidence (researcher):** high · **verdict:** (unmatched)
- **detail:** RFC 5321/5322 atext permits both ' and - in local parts (hyphen not at start/end), so o'brien@ is syntactically valid, but verification tools and practitioners report punctuation-stripped forms dominate because special characters cause typo and delivery problems. For hyphenated surnames, observed practices split between smushing (jsmithjones@) and keeping the hyphen (mwilliams-smith@); some companies also use only the first component of the compound surname. There is no data suggesting apostrophe-preserved addresses are common.
- **→ implication for mailmetero:** Candidate expansion order for compound/punctuated surnames: (1) punctuation-stripped concatenation, (2) hyphen-preserved (surnames only), (3) first-component-only surname (smith@ for Smith-Jones), (4) apostrophe-preserved last. Each variant multiplies the pattern set, so cap expansion (e.g., max 2 surname variants × top patterns) and let per-domain learning collapse the ambiguity after the first verified hit.

### 6. CJK and other surname-first cultures require name-order-reversal candidates and romanization variants: Chinese and Korean names are conventionally written surname-first even in English contexts (Xi Jinping), while Koreans working in Western companies usually flip to given-first, and Japanese usage is mixed — so an input 'first name, last name' cannot be trusted for these names.
- **confidence (researcher):** high · **verdict:** (unmatched)
- **detail:** Chinese names in English publications keep original order (surname first); Korean names likewise in publications, but Koreans living/working in Western countries usually adopt Western order; Japanese names have traditionally been reversed to given-first in English (though official Japanese government style since 2020 prefers surname-first, e.g., TAKAICHI Sanae). Romanization adds variance (pinyin standard for Chinese, but multiple Korean romanizations of the same surname: Lee/Yi/Rhee, Jeong/Chung/Jung). Additionally, Chinese given names are often two syllables that may appear joined, hyphenated, or spaced (Jinping / Jin-Ping / Jin Ping), and many Chinese professionals at Western firms use …
- **→ implication for mailmetero:** When the name is detected as likely CJK (surname dictionary lookup: Wang/Li/Zhang/Kim/Park/Lee/Tanaka/Sato etc., or CJK script input), mailmetero must generate both name-order interpretations and syllable-joining variants, and down-weight overall confidence to reflect the ambiguity; the LinkedIn-sourced 'preferred name' (often an adopted English name) should be treated as the primary given-name signal when present.

### 7. Nickname/diminutive resolution (Bob↔Robert, Liz↔Elizabeth) is a solved data problem via open-source lookup tables — carltonnorthern/nicknames (~1,100 canonical US given names with diminutives, Python/TypeScript/SQL bindings, permissive license) and diminutives.db — and matters most exactly where first@ formats dominate (small companies).
- **confidence (researcher):** high · **verdict:** (unmatched)
- **detail:** Multiple maintained GitHub datasets map formal names to diminutives bidirectionally: carltonnorthern/nicknames (hand-curated CSV, ~1,100 canonical names), HaJongler/diminutives.db (male/female diminutive CSVs), plus derivative packages (yooper/nicknames on Packagist, TylerRick/nick_names). Since first@ is the top format at sub-50-person companies (42-71% per Prospeo) and LinkedIn display names are often the informal form ('Bob Smith') while IT provisioned the formal one (robert@) or vice versa, both directions of the mapping are needed.
- **→ implication for mailmetero:** Vendor one of these CSVs (carltonnorthern/nicknames is the best maintained) into mailmetero as a Postgres table, expand candidates with the canonical form AND top diminutives at reduced weight, and boost nickname-variant candidates specifically when the company is small or the learned pattern is first@ — this is a cheap accuracy win BounceZero's science can be extended with.

### 8. Name collisions at a company are resolved almost universally by middle-initial insertion first (john.m.smith@) and numeric suffix second (jsmith2@, john.smith2@) — a direct consequence of Active Directory / IT provisioning conventions — and stale addresses are (correctly) never reused for new same-name hires.
- **confidence (researcher):** medium · **verdict:** (unmatched)
- **detail:** AD naming-convention guidance and identity-management practice: on duplicate, add middle initial; if still colliding, append a number; large companies sometimes switch to employee-ID usernames entirely. Prospeo's guidance matches: 'middle initial insertion is the cleanest fallback', numeric suffix seen as less professional but common. Collision probability rises with company size and name commonness (John Smith at a 10k-employee firm is near-certain to be suffixed), meaning the naively-derived first.last@ candidate can be deliverable-but-wrong-person or taken by a different John Smith.
- **→ implication for mailmetero:** For common name pairs (maintain a name-frequency table, e.g., US Census surnames + SSA given names) at large companies, mailmetero should (a) emit middle-initial and numeric-suffix candidates when a middle name is known or the collision risk is high, and (b) attach a 'collision risk' flag to the response — an SMTP-deliverable first.last@ at a 10k-person company may reach the wrong John Smith, which affects how confidently the API should present even a verified hit.

### 9. Role mailboxes are formally standardized by RFC 2142 (info@, sales@, support@, marketing@, abuse@, postmaster@, security@, noc@, webmaster@, case-insensitive) and plus-addressing/subaddressing by RFC 5233 (everything between + and @ ignored), supported by Gmail, Outlook.com since 2013, and Microsoft 365/Exchange since late 2020 — both must be first-class concepts in derivation output, never person-guess candidates.
- **confidence (researcher):** high · **verdict:** (unmatched)
- **detail:** RFC 2142 defines the canonical role-account set and mandates case-insensitive recognition; abuse@ and postmaster@ are required for certified senders. Role accounts are shared inboxes, frequently list-bombed/spam-trapped, and penalized by outreach platforms, so returning info@ as a 'found email' for a person is both wrong and harmful to the tenant's sender reputation. RFC 5233 subaddressing is optional per-host but supported by all major consumer and business providers (Gmail/Workspace, Outlook.com, Yahoo, M365 since ~Sept 2020; Cloudflare Email Routing added it July 2025), so user+tag@domain and user@domain are the same mailbox and must canonicalize identically.
- **→ implication for mailmetero:** mailmetero needs (1) a role-account classifier (RFC 2142 list + common extensions: hello@, contact@, hr@, careers@, admin@) that tags such addresses with type=role and excludes them from person-derivation output, returning them only via an explicit 'generic contacts' endpoint; (2) canonicalization on ingest/dedupe that lowercases the local part and strips +tags before writing to the shared pattern KB, so learned patterns aren't polluted by tagged or role addresses.

## Email Verification (SMTP/MX/catch-all)  (`verification`)

> **Verifier net assessment:** High reliability. 7 of 9 claims fully VERIFIED against primary sources (RFC 7505/5321 text, Microsoft's official NDR documentation, Render's changelog, the Reacher and disposable-email-domains GitHub repos, vendor docs, and multiple independent 2026 pricing surveys). The 2 CORRECTED claims fail only on secondary details, both in the safe direction for the architecture: (1) the VPS port-25 landscape is harsher than claimed — DigitalOcean no longer unblocks at all, Vultr approvals are unreliable, and Hetzner requires ~1 month of paid account history rather than 24-48h — which strengthens the recommendation to default to a third-party verification API with an optional Hetzner probe node; (2) MillionVerifier's 1M-credit price has dropped to $359 (~$0.00036/email), making the API-fallback economics better than claimed. No claim was refuted; the core engineering conclusions (typed DNS enum with Null-MX short-circuit, four-state + sub_status response schema, same-session catch-all probing with a shared domain knowledge base, async greylist retry from a stable IP, provider fingerprinting with M365 de-weighting, cheapest-first pipeline ordering, and never probing from Render dynos) are all sound to build on.


### 1. MX resolution has three distinct outcomes that must be handled separately: explicit MX records, the implicit-MX A/AAAA fallback (RFC 5321 §5.1), and Null MX 'MX 0 .' (RFC 7505) which is an authoritative 'this domain accepts no mail' signal.
- **confidence (researcher):** high · **verdict:** (unmatched)
- **detail:** Per RFC 5321 §5.1, if a domain returns an empty MX list, the address is treated as if it had an implicit MX with preference 0 pointing at the domain's A/AAAA host — so 'no MX record' alone does NOT mean undeliverable. Per RFC 7505, a single MX record with preference 0 and a '.' exchange (Null MX) formally declares the domain accepts no mail and suppresses the A/AAAA fallback; a verifier that sees it can immediately return 'undeliverable' with no SMTP connection. A domain with no MX, no A/AAAA (or NXDOMAIN) is definitively invalid. Parked domains frequently have A records but no real mail service, which is why implicit-MX hosts deserve lower prior confidence than explicit MX.
- **→ implication for mailmetero:** mailmetero's DNS stage should return a typed enum (EXPLICIT_MX / IMPLICIT_MX_FALLBACK / NULL_MX / NO_MAIL_HOST) stored per-domain in Neon Postgres with a TTL cache; NULL_MX and NO_MAIL_HOST short-circuit the whole pipeline for free (no SMTP, no paid API credit), and IMPLICIT_MX_FALLBACK should cap the confidence score since it is often a parked domain.

### 2. SMTP RCPT TO replies map to a small, concrete interpretation table, and the correct output vocabulary is at least four states — deliverable / undeliverable / risky / unknown — not valid/invalid.
- **confidence (researcher):** high · **verdict:** (unmatched)
- **detail:** After connect + EHLO + MAIL FROM, the RCPT TO reply is the core signal: 250 (or 251) = server accepts the mailbox (deliverable, unless the domain is catch-all); 550 5.1.1 'User unknown' = mailbox does not exist (high-confidence undeliverable — this is what Google Workspace, Proofpoint, Mimecast, IronPort typically give honestly); 550 5.4.1 'Recipient address rejected: Access denied' = Microsoft 365 directory-based edge blocking (see separate finding); 552/452 = mailbox full or too many recipients (risky); 450/451 4.7.1 = temporary policy deferral, usually greylisting (unknown, retry later); 421 = service unavailable / connection throttled (unknown). Industry sources are explicit that …
- **→ implication for mailmetero:** Design the mailmetero API response schema now with status ∈ {deliverable, undeliverable, risky, unknown} plus a sub_status (e.g. invalid_mailbox, accept_all, greylisted, disabled, full_mailbox, provider_unverifiable) and the raw SMTP code/enhanced code — this matches what customers expect from ZeroBounce-class competitors and prevents a breaking schema change later.

### 3. Catch-all (accept-all) detection requires probing a guaranteed-nonexistent random local-part on the same domain, and a 250 for that probe must downgrade every positive result on the domain to 'risky', not 'deliverable'.
- **confidence (researcher):** high · **verdict:** (unmatched)
- **detail:** A catch-all server returns 250 OK to RCPT TO for any local-part, real or fabricated, so a 250 on the target address carries no information by itself. The standard technique is to send a second RCPT TO for a random string (e.g. UUID-like) local-part: if that also gets 250, the domain is accept-all. Complications documented in 2025-era guides: some servers detect the connect/EHLO/MAIL FROM/RCPT TO/QUIT probe pattern and switch to generic acceptance or blocking mid-session, so professional verifiers maintain per-domain intelligence databases (historical behavior) to supplement live probes, and some randomize probe ordering and reuse one connection for both RCPT TOs to keep the comparison …
- **→ implication for mailmetero:** mailmetero's multi-tenant DB is a genuine advantage here: persist per-domain behavior (is_catch_all, last_probed_at, probe_reply_codes, provider fingerprint from MX hostname) in a shared knowledge-base table so one tenant's probe results benefit all tenants and repeated probing of the same domain (which burns IP reputation) is avoided; do the target RCPT and the random-local-part RCPT in the same SMTP session.

### 4. Greylisting (451/450 4.7.1) and tarpitting make synchronous one-shot verification structurally impossible for a meaningful fraction of domains — retries must be scheduled minutes-to-hours later, from the SAME source IP.
- **confidence (researcher):** high · **verdict:** (unmatched)
- **detail:** Greylisting records a triplet of (sender IP, MAIL FROM, RCPT TO) and temp-rejects the first attempt with 450/451 (commonly 451 4.7.1); a retry from the same triplet after the greylist window (typically 5-15 minutes, sometimes longer) succeeds. Because the triplet includes the source IP, rotating IPs between attempt and retry defeats your own retry. Tarpitting servers deliberately delay responses (30s+ per command) to make automated probing uneconomical, so per-connection timeouts must be generous on flagged domains without stalling the whole worker pool. Best practice from verifier vendors: honor 4xx with backoff over hours, return 'unknown/greylisted' provisionally, and background re- …
- **→ implication for mailmetero:** This directly validates mailmetero's planned Render background-worker + Neon architecture: verification must be an async job with a retry queue keyed on (domain, address, source IP) honoring greylist intervals, per-MX-host rate limits and adaptive timeouts stored in Postgres; the synchronous REST endpoint should return 'unknown - pending retry' with a job ID or webhook rather than blocking.

### 5. Microsoft 365 / Exchange Online is effectively not RCPT-verifiable: its 550 5.4.1 comes from Directory-Based Edge Blocking (DBEB) and is only emitted when DBEB is enabled and directory data is fresh; many tenants behave as accept-all, and O365 aggressively blocks probe-patterned connections.
- **confidence (researcher):** high · **verdict:** (unmatched)
- **detail:** Microsoft documents 550 5.4.1 'Recipient address rejected: Access denied' as DBEB rejecting mail at the edge for recipients not present in the tenant directory — but Microsoft's own troubleshooting notes stale/incomplete directory sync (up to 24h lag) and misconfigured accepted-domain types (Authoritative vs Internal relay) as causes of false rejections, meaning 550 5.4.1 can fire for VALID addresses and be absent for invalid ones (tenants on Internal relay or without DBEB accept everything at RCPT time). Independent 2025-era practitioner sources state flatly that Outlook/Office 365 (plus Gmail, Proofpoint, Mimecast in varying degrees) 'either always return 250 regardless of the mailbox, …
- **→ implication for mailmetero:** mailmetero should fingerprint the provider from MX hostnames (*.mail.protection.outlook.com → M365; aspmx.l.google.com → Workspace; *.pphosted.com → Proofpoint; *.mimecast.com etc.), stored in the shared domain KB, and for M365 domains skip or de-weight SMTP evidence entirely — score from pattern priors (77% first.last) and treat a 550 5.4.1 as a moderate (not definitive) negative signal; for Google/Proofpoint/Mimecast/IronPort treat 550 5.1.1 as strong negative evidence.

### 6. Outbound port 25 is blocked on Render's free tier (all SMTP ports 25/465/587, fully enforced since Sept 26, 2025) and on AWS EC2 by default; DigitalOcean/Vultr unblock on request, Hetzner opens it after an initial anti-abuse window — so mailmetero cannot rely on live SMTP from its Render host.
- **confidence (researcher):** high · **verdict:** (unmatched)
- **detail:** Render's changelog confirms free web services block outbound 25/465/587 across all regions since Friday Sept 26, 2025; paid instance types are exempt from that block, but even on paid tiers egress goes out shared cloud IPs with no rDNS control, which receiving MTAs score poorly (many reject or tarpit connections from cloud ranges regardless of port availability). Provider landscape for a dedicated probe host: AWS blocks EC2 port 25 by default and has become extremely restrictive about removal; DigitalOcean blocks on droplets created after Oct 2019 but unblocks selectively via ticket; Vultr blocks by default and is known to approve reasonable requests in ~24-48h; Hetzner allows port 25 …
- **→ implication for mailmetero:** Architect verification as a pluggable backend behind one interface: (a) default = third-party verification API callable over HTTPS/443 from Render, (b) optional 'probe node' = a $5-10/mo Hetzner or Vultr VPS with static IP, FCrDNS, and matching HELO hostname running an internal SMTP-probe microservice that the Render worker calls over authenticated HTTPS; never attempt raw RCPT probing from Render dynos themselves.

### 7. Third-party verification API costs (2026) range roughly $0.0005-$0.008 per check — MillionVerifier ~$0.00055/email at 1M volume ($37/10k PAYG, credits never expire) vs ZeroBounce/NeverBounce at $0.008 list price ($0.0035-0.004 at 100k volume) — making API fallback economically viable as mailmetero's default backend.
- **confidence (researcher):** medium · **verdict:** (unmatched)
- **detail:** 2026 pricing surveys: ZeroBounce core API $0.008/request (min purchase 2,000 credits/$20; $0.0035 at 100k; $99/mo for 25k monthly via ZeroBounce ONE); NeverBounce $0.008 up to 10k, $0.004 at 100k; MillionVerifier $549 per 1M (=$0.000549 each), $37/10k pay-as-you-go, no expiry, no monthly fee. Aggregate market tiering: MillionVerifier $0.50-1.50 per 1k, mid-tier (Clearout, DeBounce) $1-3/1k, premium (ZeroBounce, NeverBounce, Kickbox) $4-10/1k. Note these vendors themselves face the same M365/catch-all limits, so paying does not buy certainty on accept-all domains.
- **→ implication for mailmetero:** mailmetero's quota/billing model can comfortably absorb a wholesale cost of ~$0.001-0.004 per verification by caching results in the shared Neon KB (verify each unique address once per e.g. 30-90 days across ALL tenants) and only calling the paid API for cache misses after free local checks (syntax, DNS/null-MX, disposable, catch-all-known) have filtered the input.

### 8. reacherhq/check-if-email-exists is the leading open-source self-hosted verifier (Rust, Dockerized HTTP backend) but it hard-requires open outbound port 25 and is AGPL-3.0/commercial dual-licensed — a legal constraint for a hosted SaaS.
- **confidence (researcher):** high · **verdict:** VERIFIED
- **detail:** The project performs the full pipeline (syntax, MX, SMTP RCPT probe, catch-all detection, disposable detection, provider-specific handling incl. headless checks for some providers) without sending mail, ships an HTTP backend Docker image suitable for running on a VPS, and its README states it will simply hang/time out where outbound 25 is blocked, recommending 'a Linux cloud server with a static IP and no blocked ports'. Licensing is dual: AGPL-3.0 for AGPL-compatible open-source use, otherwise a paid commercial license from Reacher — running it as a backend of a closed-source hosted service triggers the AGPL network clause or requires the commercial license.
- **→ implication for mailmetero:** For the self-hosted probe-node option, either (a) budget for Reacher's commercial license, (b) open-source the probe microservice under AGPL-compatible terms while keeping the mailmetero API proprietary and talking to it over HTTP (still legally gray — get the commercial license if in doubt), or (c) write a small in-house prober reusing BounceZero's proven SMTP logic, which the user already owns outright — likely the cleanest path.

### 9. ### 3. The website domain and the email domain are frequently different entities, and a live MX lookup is the only cheap ground-truth test that a candidate domain actually receives mail — a domain with no MX records (or an RFC 7505 null MX) cannot be an employee email domain regardless of how good the name match looks.
- **confidence (researcher):** high · **verdict:** (unmatched)
- **detail:** Marketing sites sit on vanity TLDs (.io/.ai/.xyz), landing-page domains, or regional storefronts while mail runs on the canonical corporate domain; post-acquisition companies keep old mail domains with forwarding for extended periods, or run permanent subsidiary domains distinct from the parent brand's site. - **→ implication for mailmetero:** mailmetero's resolver pipeline must be: candidate domain(s) → MX lookup → drop/downgrade any candidate with no MX → prefer the candidate whose MX points at a business mail stack. Store MX results (with TTL-based expiry, e.g. re-check after 30-90 days) in the shared Postgres knowledge base; DNS-over-HTTPS (Google/Cloudflare resolvers) works from Render even though port 25 is blocked, since MX lookups are DNS, not SMTP.

### 4. MX hostname fingerprinting reliably identifies the mail provider — aspmx.l.google.com = Google Workspace, *.mail.protection.outlook.com = Microsoft 365, *.pphosted.com = Proofpoint, *.mimecast.com = Mimecast, *.iphmx.com = Cisco IronPort — and among the Tranco top-1M domains over 60% of mail is on Google (41%) + Microsoft (20%), so two fingerprints cover the majority of targets.
- **confidence (researcher):** high · **verdict:** (unmatched)

### 8. Companies routinely operate multiple legitimate email domains simultaneously — regional ccTLD variants, retained subsidiary domains after M&A, and old domains kept alive via forwarding during rebrands — so a one-company-one-domain data model is wrong.

### 9. ### 3. Meta v. Bright Data (N.D. Cal., Jan 23, 2024) held that LOGGED-OFF scraping of public data did not breach Meta's ToS because a logged-out scraper is not a 'user' — but this is a contract-drafting artifact, not a rule that public scraping is legal, and it has limited carryover to LinkedIn.

- **→ implication for mailmetero:** The only even arguably defensible collection zone is logged-out, truly public, non-LinkedIn data (e.g., company websites, DNS/MX records, press pages). mailmetero's data-sourcing policy should draw the line there: DNS/MX lookups, homepage crawls of the target company's own domain, and user-provided inputs — never authenticated-wall content from any platform.

### 4. - **confidence (researcher):** high · **verdict:** (unmatched)

### 5. ### 6. - **confidence (researcher):** high · **verdict:** (unmatched)

- **→ implication for mailmetero:** mailmetero can legitimately serve UK/US B2B use cases, but should expose per-jurisdiction guidance and let tenants tag campaigns by recipient country; classify sole-trader domains cautiously; and treat the 35x-larger UK fine ceiling as a reason to make the suppression/objection machinery first-class rather than best-effort.

### 7. CAN-SPAM is an opt-out regime — cold B2B email is legal in the US without prior consent — but each non-compliant email carries a civil penalty of up to $53,088 (FTC inflation adjustment effective Jan 17, 2025), and liability reaches both the sender and the company whose product is promoted.

- **→ implication for mailmetero:** mailmetero itself (finder + verifier) sits outside CAN-SPAM's direct scope, but its customers are squarely inside it — so provide a per-tenant and global suppression-list API, export flags marking opted-out addresses, and acceptable-use terms requiring CAN-SPAM compliance, which also insulates mailmetero from 'initiator/facilitator' theories.

### 9. The defensible 2026 compliance envelope for an email-finder service is settled enough to specify concretely: derivation-only sourcing, no authenticated-wall data, published transparency notice, working DSAR + absolute marketing-objection suppression, short person-level retention, and company-level-only shared learning — while the legality of logged-out public scraping and non-9th-Circuit CFAA scope remain genuinely unsettled.

## Service Engineering (Neon + Render)  (`service-arch-neon-render`)

> **Verifier net assessment:** Highly reliable: 9/9 claims VERIFIED against primary sources (Render changelog/docs, Neon docs fetched directly, GitHub issues/repos, the cited case studies). Every specific number checked — PgBouncer max_client_conn=10000 and 0.9×max_connections pool size, 5-min autosuspend with few-hundred-ms cold start, 100 CU-hours free (doubled Oct 2025), 15-min Render spin-down with ~1-min (claim said 30-60s) spin-up, $7/$25 instance pricing, cron single-active-run guarantee, and the 946ms bcrypt case study — matched current 2025-2026 sources. Notably, Neon's own choose-connection doc names Render explicitly as a platform where standard TCP drivers beat the serverless driver, directly confirming claim 6. Only two trivial softenings: render.yaml root location is a customizable default, and the serverless-driver per-query latency figures trace to one third-party benchmark (direction confirmed by Neon docs, exact ms not independently re-measured). The architectural implications drawn from these facts (no port 25 on Render on any plan; both pooled and unpooled DATABASE_URLs; statement_cache_size=0; poll backoff to respect autosuspend) are sound and can be built on as-is.


### 1. Live SMTP RCPT verification is impossible from Render on ANY plan: outbound port 25 is permanently blocked for all services (free and paid), and since September 26, 2025 free-tier services also block ports 465/587.
- **confidence (researcher):** high · **verdict:** (unmatched)
- **detail:** Render runs on Amazon EC2, which blocks outbound port 25 platform-wide; this applies to web services, background workers, and cron jobs on every instance type. Ports 465/587 (submission ports for sending mail via a relay) work only on paid instances. RCPT-based verification specifically requires connecting to arbitrary MX hosts on port 25 — there is no paid tier or support ticket that unblocks it on Render.
- **→ implication for mailmetero:** mailmetero's verifier tier must be designed around this: (a) integrate a third-party verification API (e.g., a bulk verifier with per-check pricing) behind a provider interface, and/or (b) run an optional self-hosted 'SMTP probe' microservice on a VPS with port 25 open (Hetzner/OVH with rDNS) that the Render worker calls over HTTPS. The core product must degrade gracefully to pattern+MX+provider-fingerprint confidence scoring when no probe is available — exactly matching the BounceZero insight that M365 is unverifiable anyway.

### 2. Neon exposes two connection strings per branch — pooled (hostname contains '-pooler', PgBouncer in transaction mode, max_client_conn=10000, default_pool_size=0.9×max_connections) and direct — and pooled connections do not support SET/RESET, LISTEN/NOTIFY, session-level advisory locks, or SQL-level PREPARE.
- **confidence (researcher):** high · **verdict:** (unmatched)
- **detail:** Neon's built-in PgBouncer runs pool_mode=transaction with query_wait_timeout=120s; per-(user,database) pools share the compute's max_connections (104–4000 depending on CU size). Protocol-level prepared statements ARE supported (PgBouncer ≥1.22), but SQL-level PREPARE/DEALLOCATE, session state (SET), LISTEN/NOTIFY, WITH HOLD cursors, and session-level advisory locks are not. Migrations, pg_dump, and anything needing session semantics must use the direct (unpooled) string; sslmode=require&channel_binding=require is the standard suffix on Neon connection strings.
- **→ implication for mailmetero:** mailmetero needs BOTH strings in config: DATABASE_URL (pooled) for the FastAPI app, DATABASE_URL_UNPOOLED (direct) for Alembic migrations and for the background worker if it uses LISTEN/NOTIFY job wakeups. Rate limiting must NOT rely on session-level advisory locks over the pooled string — use transaction-level advisory locks (pg_advisory_xact_lock) or plain atomic UPDATE counters instead.

### 3. asyncpg through Neon's pooled endpoint hits the classic PgBouncer prepared-statement trap (DuplicatePreparedStatementError '__asyncpg_stmt_N__ already exists'); the standard fix is statement_cache_size=0 (SQLAlchemy: connect_args={'statement_cache_size': 0} plus NullPool or a unique prepared_statement_name_func), or use psycopg3 which handles it more gracefully.
- **confidence (researcher):** high · **verdict:** (unmatched)
- **detail:** asyncpg auto-prepares and caches up to 100 statements per connection, but PgBouncer transaction mode hands the same backend to different clients, so cached statement names collide. On managed poolers like Neon where you can't tune PgBouncer config yourself, disabling asyncpg's cache is the documented fix; psycopg3 (async) with default prepare_threshold behavior is the commonly cited long-term alternative. Note Neon's PgBouncer does support protocol-level prepared statements (1.22+ max_prepared_statements), which mitigates but has not eliminated reported asyncpg collisions in practice.
- **→ implication for mailmetero:** Pin this in mailmetero's DB setup from day one: SQLAlchemy 2.0 async engine with asyncpg and connect_args={'statement_cache_size': 0} when pointed at the -pooler host (or route the app through the direct string with SQLAlchemy's own pool, sized well under Neon's max_connections). Document the error string in the runbook — it is the single most common Neon+FastAPI production bug.

### 4. Neon compute autosuspends after 5 minutes of inactivity with a ~300–500ms cold start on wake; the free plan gives 100 CU-hours/month per project (doubled from 50 in Oct 2025), enough for ~400 hours of a 0.25-CU compute.
- **confidence (researcher):** high · **verdict:** (unmatched)
- **detail:** Scale-to-zero is always on for free plan (5-min idle timeout, not configurable there; configurable/disable-able on paid). Cold start adds a few hundred ms to the first query after suspension — tolerable for an API, but a polling worker will keep the compute permanently awake and burn CU-hours 24/7. Neon repriced substantially after the May 2025 Databricks acquisition (usage-based, no minimums).
- **→ implication for mailmetero:** The background worker's poll loop dictates Neon cost: a naive 1-second SELECT poll keeps compute awake forever (~720 CU-hours/month at 1 CU — over free tier). mailmetero's worker should poll with generous backoff when the jobs table is empty (e.g., 30–60s idle interval) or be woken via an API-side signal, and the app must treat a first-query latency spike of ~0.5s as normal, not a timeout.

### 5. A render.yaml Blueprint can declare the whole mailmetero topology — 'web' service (with healthCheckPath), 'worker' service (no port), and 'cron' jobs (schedule required; Render guarantees at most one concurrent run per cron job) — plus envVarGroups shared across services; Render free web services spin down after 15 minutes idle (~30–60s cold start) and free tier has no background workers.
- **confidence (researcher):** high · **verdict:** (unmatched)
- **detail:** Blueprint file must be render.yaml at repo root; service types include web, worker, cron, pserv. Env groups let web+worker share DATABASE_URL and API secrets; existing env vars are preserved if omitted from the Blueprint. Free web instance = 512MB/0.1 CPU with 750 free instance-hours/workspace/month; Starter is $7/mo (512MB/0.5 CPU), Standard $25/mo (2GB/1 CPU). Background workers require a paid plan. Cron jobs' single-active-run guarantee is a free idempotency win for scheduled cleanup tasks.
- **→ implication for mailmetero:** Ship mailmetero as one render.yaml: web (FastAPI, healthCheckPath: /healthz doing a cheap DB ping), worker (bulk verification consumer, Starter $7), cron (nightly TTL cache purge + quota reset). Minimum realistic production cost ≈ $14/mo (Starter web + Starter worker) + ~$0–5 Neon. Don't demo on free tier if API latency matters — 30–60s spin-up will look broken to users.

### 6. The @neondatabase/serverless HTTP/WebSocket driver is the wrong choice for a long-lived Render service; a standard TCP driver (asyncpg/psycopg/node-postgres) with a connection pool is strictly better there — the serverless driver exists for edge/lambda environments and adds proxy overhead (HTTP ~15–30ms, WebSocket ~5–15ms vs TCP ~3–10ms per query).
- **confidence (researcher):** high · **verdict:** (unmatched)
- **detail:** Neon's own docs say if your app runs on a long-lived server you should use a normal Postgres driver; the serverless driver's HTTP mode is for one-shot queries from environments without TCP, and its WebSocket mode emulates pg for edge runtimes. Multi-statement transactions are dramatically slower over the proxy (a 10-statement tx: ~30ms TCP vs 80–150ms WebSocket).
- **→ implication for mailmetero:** Since mailmetero runs on Render (long-lived containers, full TCP), use SQLAlchemy 2.0 async + asyncpg (or psycopg3) with the app's own pool — do not cargo-cult the Neon serverless driver from Vercel tutorials. This also frees mailmetero to use interactive transactions cheaply (job claiming, quota decrement + insert in one tx).

### 7. For bulk-verification jobs, a Postgres-native queue using FOR UPDATE SKIP LOCKED (raw SQL or a library like Procrastinate/PgQueuer) beats Celery/Redis for this product: exactly-once claiming, transactional enqueue with the API write, and zero extra infrastructure on Render.
- **confidence (researcher):** high · **verdict:** (unmatched)
- **detail:** SELECT ... FOR UPDATE SKIP LOCKED lets N workers atomically claim jobs without blocking or double-processing; Python libraries built on it (Procrastinate — SQLAlchemy/psycopg-based, mature; PgQueuer — asyncpg-based with per-entrypoint concurrency limits) are production-used in 2025. Celery would force adding a Redis instance (~$10+/mo on Render Key Value) and loses transactional enqueue (job row committed atomically with the API request row). Caveat from finding #2: LISTEN/NOTIFY wakeups only work over Neon's DIRECT connection, and constant polling fights Neon autosuspend — use a modest poll interval with backoff from the worker's single direct connection.
- **→ implication for mailmetero:** Architecture: API inserts into a jobs table (status queued/claimed/done, attempts, locked_by, locked_at) in the same transaction as the bulk-request record; the Render worker claims batches via SKIP LOCKED over the direct Neon connection, processes candidate emails concurrently with asyncio semaphores, and writes results back. Add a cron sweep for jobs stuck in 'claimed' past a visibility timeout. No Redis, no broker, one database.

### 8. API keys should be verified with a fast keyed hash (HMAC-SHA256 with a server-side pepper, constant-time compare), NOT bcrypt/argon2 — bcrypt on every request is a documented ~100–950ms/request tax; security comes from the key's 256 bits of entropy, with a short stored prefix (e.g. 'mm_live_abc123…') for lookup and display.
- **confidence (researcher):** high · **verdict:** (unmatched)
- **detail:** 2025 consensus: generate 32 random bytes (secrets.token_urlsafe), show once, store SHA-256/HMAC-SHA256 digest + first ~8 chars as an indexed lookup prefix; verify by recomputing the digest and comparing with hmac.compare_digest. Slow memory-hard hashes remain correct for human passwords (Argon2id per OWASP) but are a per-request latency/CPU disaster for stateless API auth — one migration case study measured 946ms/request spent in bcrypt alone.
- **→ implication for mailmetero:** mailmetero's api_keys table: id, tenant_id, key_prefix (indexed), key_hash (HMAC-SHA256 with APP_PEPPER env secret), scopes, created_at, revoked_at, last_used_at. Auth middleware does one indexed lookup by prefix + constant-time digest compare — microseconds, no CPU spike at 512MB/0.5-CPU Render instances. Keep Argon2id only for the web UI's human login, if any.

### 9. Recommended concrete stack (2025–2026 best practice): FastAPI + SQLAlchemy 2.0 async + asyncpg + Alembic on Render web, a plain-Python asyncio worker on a Render worker service consuming a SKIP LOCKED jobs table, with quotas as atomic UPDATE counters and idempotency via a UNIQUE(tenant_id, idempotency_key) table — all on one Neon database.
- **confidence (researcher):** medium · **verdict:** (unmatched)
- **detail:** FastAPI remains the dominant async Python API framework in 2025-2026 production guides (typed request/response models via Pydantic v2, OpenAPI for the public API for free — important for a developer-facing service); SQLAlchemy 2.0 async + asyncpg is the standard pairing, with Alembic migrations run against the UNPOOLED Neon URL in Render's preDeployCommand. Rate limiting: per-tenant token-bucket or fixed-window counters as a single atomic UPDATE ... SET used = used + 1 WHERE used < quota RETURNING (no advisory-lock session state needed, pooler-safe); optionally an in-process SlowAPI limiter for burst control. Verification-result cache: a results table keyed on (email) with verified_at + …
- **→ implication for mailmetero:** This is the build plan: repo with app/ (FastAPI), worker/ (queue consumer sharing the same models package), alembic/, render.yaml declaring web+worker+cron and an env group (DATABASE_URL pooled, DATABASE_URL_UNPOOLED, APP_PEPPER, verification-API key). It deliberately reuses BounceZero's science modules as an installable core library while everything stateful (tenants, keys, quotas, jobs, learned patterns, cached verdicts) lives in Neon — the exact differentiation from the offline package. Confidence 'medium' only on stack choice being 'best' (Litestar/Go are viable alternates); each component's facts are individually high-confidence.
